1. Project Overview

SHAH (SecondHand Auction Hub) is a web-based auction platform built with ASP.NET Core 8 MVC. It enables users to list used items for auction, place manual and proxy (auto) bids in real-time, process payments via Stripe, and leave reviews after completed transactions. The platform includes 4 implemented algorithms — Proxy Bidding (economic mechanism), Dynamic Bid Increment, Time Extension (anti-sniping), and Item Recommendation (popularity + recency scoring) — alongside real-time updates via SignalR, scheduled auction lifecycle via Hangfire, and in-app notifications.

4. Algorithms — Detailed Documentation

Algorithm 1: Proxy (Auto) Bidding Major

Type: Major — Economic auction mechanism (Vickrey / second-price variant)
File: DataAccess/Repository/AutoBidRepository.cs
Called from: Controllers/BidController.cs:PlaceBid(), Controllers/AutoBidController.cs:SetAutoBid()
Academic Backing: Roth & Ockenfels (Stanford/MIT), American Economic Review — proxy bidding on eBay
Lines of Code: ~130 lines (ProcessAutoBidsAsync + NotifyAndDeactivateAsync)

What It Does

Allows a user to set a maximum bid amount. The system automatically bids on their behalf, incrementally, up to that maximum — but only as needed to maintain their position as the highest bidder. The winner pays the second-highest maximum + one increment (eBay-style proxy), not their full maximum.

How It Works (Step-by-Step)

  1. Load all active auto-bids for the auction in a single query, sorted by MaxAmount descending then CreatedAt ascending.
  2. Calculate minimum required = currentHighestBid + DynamicBidIncrement(currentHighestBid).
  3. Filter challengers — auto-bids from users who are NOT the current highest bidder AND whose MaxAmount >= minRequired.
  4. No challengers? Deactivate all exhausted auto-bids (those with MaxAmount < minRequired), send notifications, and exit.
  5. Best challenger = first in sorted list (highest MaxAmount, then earliest created).
  6. Compare challenger vs current holder's auto-bid (if any):
  7. • If challenger's MaxAmount > holder's MaxAmount: challenger wins at min(challenger.MaxAmount, holder.MaxAmount + increment). Holder is deactivated.
    • If holder's MaxAmount ≥ challenger's MaxAmount: holder stays at min(holder.MaxAmount, challenger.MaxAmount + increment). Challenger is deactivated.
    • If holder has no auto-bid: challenger wins at min(challenger.MaxAmount, currentHighestBid + increment).
  8. Deactivate all auto-bids whose MaxAmount is below the winning amount (they can no longer compete).
  9. Place one winning bid with the determined amount and winner.
  10. Single SaveChangesAsync — all changes (deactivations + new bid) committed atomically.
  11. Check time extension — if the proxy bid triggered the anti-sniping threshold, extend the auction.

Edge Cases Handled

  • No active auto-bids at all → early exit
  • Only current holder has auto-bid, no challengers → no changes needed
  • All auto-bids exhausted below minimum required → deactivate all, notify all
  • Challenger has same MaxAmount as holder → holder wins (earliest-created tiebreak)
  • Same user already has auto-bid → update MaxAmount instead of creating duplicate
  • Auto-bid deactivation during proxy resolution → notification sent immediately

Notification Integration

When auto-bids are deactivated (exhausted or outbid), NotifyAndDeactivateAsync() creates in-app notifications: "Your auto-bid on {item} has been stopped — you've been outbid." Notifications are batched — fetches auction/item info once for all deactivated users.

Algorithm 2: Dynamic Bid Increment Minor

Type: Minor — Percentage-tiered calculation
File: Utility/BiddingEngine.cs
Used by: BidController.cs, AutoBidController.cs, AutoBidRepository.cs
Lines of Code: ~35 lines

What It Does

Calculates the minimum amount a bidder must exceed the current highest bid by. Uses 6 tiers so that low-value items grow quickly but high-value items don't require absurdly large jumps. Replaces a flat $1 increment (the prior system) with fair, proportional scaling.

Tier Breakdown

TierCurrent Bid RangeIncrement %Example
1Below $1005%$50 bid → $2.50 increment
2$100 – $4994%$200 bid → $8.00 increment
3$500 – $1,9993%$1,000 bid → $30.00 increment
4$2,000 – $9,9992%$5,000 bid → $100.00 increment
5$10,000 – $49,9991.5%$20,000 bid → $300.00 increment
6$50,000+1%$100,000 bid → $1,000.00 increment

Minimum increment is always $1.00 regardless of tier.

How It Works (Step-by-Step)

  1. Receive current highest bid as input.
  2. Check which tier the bid falls into (6 conditionals).
  3. Multiply by the tier's percentage and round to 2 decimal places.
  4. Take the maximum of (calculated increment, $1.00 minimum).
  5. Return the result — caller adds this to the current highest bid to get the new minimum acceptable bid.

Edge Cases Handled

  • Zero or negative current bid → tier 1 (5%) applies, min $1
  • Micro-bids below $1 → floor to $1 minimum
  • Decimal rounding to 2 places via Math.Round

Algorithm 3: Time Extension (Anti-Sniping) Minor

Type: Minor — Anti-sniping auction rule
File: DataAccess/AuctionTimeExtension.cs
Used by: BidController.cs:PlaceBid(), AutoBidController.cs:SetAutoBid(), AutoBidRepository.cs:ProcessAutoBidsAsync()
Academic Backing: Roth & Ockenfels — sniping and late bidding on eBay
Lines of Code: ~38 lines

What It Does

Prevents last-second "sniping" (placing a winning bid in the final seconds so no one can respond). When a bid is placed within the last 5 minutes of an auction, the deadline extends by 2 minutes, giving other bidders a fair chance to respond. Up to 3 extensions per auction, with a 2-minute cooldown between extensions to prevent infinite loops.

Parameters

ParameterValue
Extension threshold5 minutes (bid must be within this window to trigger)
Extension amount2 minutes (added to EndTime)
Max extensions3 (per auction)
Cooldown2 minutes (between successive extensions)

How It Works (Step-by-Step)

  1. Check if EndTime is null → return null (no change).
  2. Calculate timeRemaining = EndTime - UtcNow.
  3. Calculate timeSinceLastExtension = UtcNow - LastExtensionTime (or MaxValue if null).
  4. Verify all 4 conditions simultaneously:
    • timeRemaining > 0 (auction hasn't ended yet)
    • timeRemaining ≤ 5 minutes (within threshold)
    • ExtensionCount < 3 (not exceeded max)
    • timeSinceLastExtension ≥ 2 minutes (cooldown satisfied)
  5. If all conditions true: add 2 minutes to EndTime, increment ExtensionCount, set LastExtensionTime.
  6. Return the new EndTime (or null if not extended).

Edge Cases Handled

  • Multiple rapid bids within threshold → cooldown prevents more than 1 extension per 2 minutes
  • Bid placed exactly at threshold boundary → exact comparison works (≤, not <)
  • Auction already extended 3 times → no further extensions
  • Bid after auction technically ended → timeRemaining ≤ 0, no extension
  • No previous extension → timeSinceLastExtension = MaxValue, always passes cooldown check

SignalR Integration

When an extension occurs, BidController sends both ReceiveBidUpdate and ReceiveAutoBidUpdate to the item's SignalR group, plus a TempData message: "Auction extended to {date} UTC due to late bid."

Algorithm 4: Item Recommendation Minor

Type: Minor — Popularity + Recency scoring
File: DataAccess/Repository/ItemRepository.cs
Called from: Controllers/HomeController.cs:Details()
Lines of Code: ~42 lines (GetRecommendedItemsAsync)

What It Does

On every auction detail page, shows up to 4 recommended items from the same category. Items are scored by a combination of bid popularity (bid count × 2) and recency (newer listings get bonus points). The highest-scored items appear first.

Scoring Formula

// Score = Popularity + Recency score = (bidCount * 2) + recencyBonus; // Recency bonus (based on days since auction started): if daysOld <= 1 → bonus = 10 elif daysOld <= 3 → bonus = 6 elif daysOld <= 7 → bonus = 3 else → bonus = 0

How It Works (Step-by-Step)

  1. Fetch all items in the same category (excluding the current item) that are Approved or InAuction status.
  2. Include Auctions and Bids in the query (one round trip to DB).
  3. For each candidate, compute:
    • bidCount = number of bids on the item's auction
    • daysOld = days since auction started
    • recencyBonus from the table above
    • score = (bidCount × 2) + recencyBonus
  4. Sort by score descending (most popular + recent first).
  5. Take the top 4 items and return them.

Edge Cases Handled

  • No other items in the same category → returns empty list
  • Item with zero bids → score = recencyBonus only (older items score 0)
  • Multiple auctions for one item → uses the first auction's start time and bid count
  • Only 1-3 items in category → returns what's available (less than 4 is fine)

Bonus: Winner Determination Algorithm Minor

File: DataAccess/Repository/IAuctionRepository.cs (EndAuctionAsync)
Purpose: Determines the winner when an auction closes

When an auction's EndTime arrives, the Hangfire job calls EndAuctionAsync(). The winner is the user who placed the highest bid. Their UserId is stored in Auction.WinnerId. The auction status changes to Sold (if bids exist) or Unsold (if no bids received). Unsold items are auto-relisted (up to 3 times). The winning bidder is notified via in-app notification.

Bonus: Schedule Rotation Algorithm Minor

File: Services/AuctionSchedulerService.cs
Purpose: Manages weekly auction cycle

After each auction cycle ends, the schedule rotates: Next → Current, and a new Next schedule is created with default values. The GetNextUtcForLocal() method calculates the next occurrence of a specific day+hour, supporting both current-week and next-week modes. Hangfire jobs are scheduled for the Start and End times, with deduplication (won't schedule if the job already exists).

8. High-Level Use Case Flow

  1. Admin sets up auction schedule (start day/hour, end day/hour).
  2. Seller (User in Seller role) lists an item with description, starting price, category, condition, image.
  3. Admin approves the item → it enters the next scheduled auction cycle.
  4. Buyer (User in Buyer role) browses items on homepage, clicks to view details.
  5. Buyer places a manual bid (or sets an auto-bid with a maximum amount).
  6. Bid is processed: increment calculated, auto-bids resolved, time extension checked, notifications sent.
  7. All connected users see live updates via SignalR (bid list, current highest, auto-bid status, min bid amount).
  8. Hangfire triggers EndAuction at the scheduled EndTime (potentially extended by anti-sniping).
  9. Winner is determined, payment status set to ToPay, seller notified.
  10. Winning bidder pays via Stripe Checkout → status becomes Paid.
  11. After payment, buyer and seller can leave reviews for each other.
  12. Unsold items are auto-relisted (up to 3 times). Schedule rotates for the next week.
